## 確認試験

以下の設問について、最も適切な選択肢を選んでください。
（※システム環境に実装される場合は、選択式の回答フォームになります）

### 設問1：一人称エンジニアについて

本コースで定義する「一人称エンジニア」の行動として、**最も不適切**なものはどれですか？

1. タスクを受け取ったら、まず作業内容を分解して段取りを組む。
2. 疑問が生じても、まずは自分で推測して実装を進め、後でまとめて報告する。（ **正解** ）
3. 提示された想定時間を超えそうな場合、着手前や早い段階で相談する。
4. 不明点があれば、質問フォーマットに沿って整理してから質問する。

### 設問2：本コースが目指すゴール

本コース「即戦力フロントエンド開発コース」の達成目標として、最も適切なものはどれですか？

1. 最新のフレームワークの機能をひたすら丸暗記すること。
2. これまでに学んだ技術（ReactやPostgreSQLなど）の基礎構文を順番に学び直すこと。
3. 既存の技術を統合し、実務レベルのシステムを一気通貫で開発する経験を積むこと。（ **正解** ）
4. コードを一切書かずに、ノーコードツールを用いてWebアプリケーションを完成させること。

### 設問3：非技術領域（ヒューマンスキル）の扱い

本コースにおける「コミュニケーション能力」や「TPOに合わせた立ち回り」といった非技術領域の考え方として、正しいものはどれですか？

1. 現場では全く重要ではないため、学ぶ必要はない。
2. 現場では非常に重要だが、テキストの学習だけで身につけるのは難しいため本コースではスコープ外としている。（ **正解** ）
3. プログラミングスキルよりも優先順位が高いため、コースの半分以上の時間を使って徹底して学ぶ。
4. 本コースを修了すれば、現場で必要な全てのヒューマンスキルが完璧に身につくよう設計されている。

### 設問4：タスク整理のフォーマット

タスクを受け取った際に情報を整理する「フォーマット」の中に**含まれない項目**はどれですか？

1. 【概要】（タスクの目的や全体像）
2. 【タスクの詳細】（具体的な手順への分解）
3. 【使用言語の歴史】（プログラミング言語の成り立ちの調査など）（ **正解** ）
4. 【確認事項】（作業開始前の不明点や質問）

### 設問5：タスク分解の粒度

タスク分解を行う際の理想的な目安として、最も適切なものはどれですか？

1. 「画面全体を作る」というような、1日〜数日かかる大きな粒度。
2. 「コードを書く」「テストする」という大ざっぱな2ステップの粒度。
3. 1つの小タスクが長くても「30分〜1時間程度」で終わる具体的な粒度。（ **正解** ）
4. 「1行ずつコードの記述内容を言語化する」といった極細の粒度。

### 設問6：時間管理の考え方

「この改修タスクは2時間でお願いします」と依頼された場合、着手前の行動として最も適切なものはどれですか？

1. すぐに実装を始め、2時間経過した時点で終わった分だけを提出する。
2. 分解した小タスクの所要時間を予測して配分し、2時間で収まるか段取りを計算する。（ **正解** ）
3. とにかく急いでタイピングし、休憩なしで2時間ぶっ通しでコードを書く。
4. 2時間では終わらないと直感で判断し、特に理由なく「4時間ください」とだけ伝える。

### 設問7：確認事項の扱い（推測実装の防止）

タスク分解の過程で「設計書には日付の表示フォーマットの記載がない」という疑問が生じました。どう対応すべきですか？

1. おそらく「yyyy/mm/dd」だろうと想像で補ってそのまま実装を進める。
2. 実装が終わった後に「フォーマットはどうしますか？」と確認する。
3. 着手前の段階で【確認事項】としてリストアップし、関係者に質問する。（ **正解** ）
4. わからない部分は実装自体をあきらめて、そのタスクを放棄する。

### 設問8：Chrome DevToolsの基本1（Console）

Chrome DevToolsの「Console（コンソール）」タブの主な役割として、正しいものはどれですか？

1. 「_API_」からのデータ取得やリクエストの成否（通信状態）を監視する。
   ※この時点では「API」という用語は分からなくてOKです。
2. 画面上の要素のHTML構造やCSSの適用状況を視覚的に検証する。
3. JavaScriptなどの実行時エラー、および `console.log` の出力を確認する。（ **正解** ）
4. データベースのテーブル構成や保存されているデータを直接編集する。

### 設問9：Chrome DevToolsの基本2（Network）

画面にデータが表示されない不具合が発生しました。
「そもそもデータがフロントエンド側に届いていないのか、それとも『_Reactのコンポーネント_』での表示処理に失敗しているだけなのか」を切り分けるために、最も適したタブはどれですか？
※この時点では「Reactのコンポーネント」という用語は分からなくてOKです。

1. Console
2. Elements
3. Network（ **正解** ）
4. Application

### 設問10：Chrome DevToolsの基本3（Elements）

「ボタンの配置が少しずれている」「文字の色が変わらない」といったUIやスタイルの不具合を調査・検証する際に、最も適したDevToolsのタブはどれですか？

1. Console
2. Elements（ **正解** ）
3. Network
4. Application

## 解答と解説

- **設問1：2 （推測実装は後戻りの原因になるためNG）**
  - 分解の過程で見つけた不明点は、着手前に必ず洗い出しておきましょう。
- **設問2：3 （既存技術の統合と実践）**
  - バラバラの知識を繋ぎ合わせ、一気通貫の開発サイクルを自ら回せる状態を目指します。
- **設問3：2 （ヒューマンスキルは現場で学ぶ）**
  - 非技術領域は実務において非常に重要ですが、テキストだけでは教えきれないため、本コースでは「情報を整理する技術（フォーマット）」に特化します。
- **設問4：3 （【使用言語の歴史】は不要）**
  - タスク整理に必要なのは「概要」「期限」「時間」「詳細」「確認事項」の5つです。
- **設問5：3 （30分〜1時間アクションが迷わない粒度がベスト）**
  - 大きすぎると途中で手が止まり、細かすぎると整理だけで時間を浪費してしまいます。
- **設問6：2 （想定内に収まるか計算する）**
  - 「どう配分すれば終わるか」を考えるのがプロの段取りです。もし収まらないと判断した場合は、着手前に「〇時間オーバーする可能性がある」と相談します。
- **設問7：3 （疑問は着手前に洗い出す）**
  - 想像や推測で実装を進める「推測実装」は、のちに大きな手戻りを生む原因になります。
- **設問8：3 （エラーが出た時はまずConsoleを見る習慣を）**
  - エラーが出たら勘で直さず、Consoleタブの赤文字エラーを読んで「事実」から解決策を探るのが鉄則です。
- **設問9：3 （通信に関する確認・問題の切り分けはNetworkタブで）**
  - リクエストがエラーになっているのか、レスポンスが空なのかを切り分けられます。
- **設問10：2 （UIやデザインの検証はElementsタブで）**
  - 適用されているCSSを確認し、その場で数値を書き換えてシミュレーションすることも可能です。
