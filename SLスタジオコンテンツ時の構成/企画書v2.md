# 企画タイトル

- 【コース企画】即戦力フロントエンド開発コース（React＋Next.js＋PostgreSQL）

# コンセプト ～ 社内研修との設計方針の違い

本コースは、既存のオフライン社内研修（2週間）の内容をSLスタジオ向けに再設計したものである。単なる研修の書き起こしではなく、**自己学習という形態に合わせて「何を残し、何を外すか」を明確に判断**した上で構成している。

## 技術面：社内研修と同等の技術スキルセットを担保

学べる技術スキルセット（Web API連携・CRUD実装・Git運用・Next.js＋Storybook開発）は社内研修とほぼ同じであり、技術面の質は担保されている。ただし、学習環境の違いに起因する以下2点を変更している。

- **DB環境：Supabase → ローカルPostgreSQL**
  - 社内研修ではオフライン実施のため講師の監視下にあり、社内で立てたSupabaseを利用しても運用上の問題がなかった。しかしSLスタジオでは不特定多数が自己学習で利用するため、同じ環境を提供・管理し続けることは現実的ではない。受講者のローカル環境で完結するPostgreSQLを採用し、データベースコースで構築済みの環境をそのまま活用する。

- **Git操作：GitHub Desktop → VSCode標準のGit機能（ソース管理タブ）**
  - 社内研修ではGitHub Desktopが主流のため導入していたが、一般的にはCLI操作やVSCode標準機能、あるいはAIによる操作が主流である。自己学習ではテキストに沿って進めるためGit操作でのエラーは起きにくく、専用GUIツールの導入よりも「差分を確認して不要コードがないか目視チェックする習慣」の定着を重視し、VSCode内で完結する標準のソース管理タブを採用した。

## 非技術分野（ヒューマンスキル）：本コースのスコープ外とする

社内研修では、報連相・TPO・ビジネスマナーといったヒューマンスキルに依存する非技術分野を講師が口頭で指導していた。しかし、これらは以下の理由からSLスタジオのテキストコンテンツとしては効果が期待しにくいと判断し、**本コースのスコープ外**とする。

- **文章化との相性が悪い。** 報連相のタイミングやTPOに応じた判断は、状況依存性が高く、テキストで網羅的に伝えることが構造的に困難である。

## ただし「開発プロセスとして必要な技術」は残す

ヒューマンスキル全般をスコープ外とする一方で、以下の要素は **開発業務に必要な「技術プロセス」** として本コースに残す。

- **開発の段取り**：設計書・要件定義書を読み解き、実装の順序を整理してから着手する技術
- **実装後の動作確認・整理**：実装後に何をしたか・何が変わったかを技術的に整理し、次のタスクに活かす技術

これらはコミュニケーション能力や人間力ではなく、 **技術的な開発プロセスの一部** であり、実践課題のChapter（Ch3以降）で 【開発の段取り】→【実装】→【動作確認】→【実装結果の整理】のサイクルを繰り返すことで実践的に定着を図る。

# ゴール(SMART)

- **As is（現状）**：「HTML・CSS初歩コース」でHTML/CSSの基本的な記述方法を学び、「Reactではじめるプログラミング基礎コース」でWebアプリが作れるようになり、「データベースコース」でDBの仕組みやPostgreSQLの操作を学び、「Git入門コース」でバージョン管理のCLI操作を習得した。しかしこれらは個別の点でしかなく、「ReactからAPIを叩いたことがない」「DBとフロントエンドを繋いだ経験がない」「既存プロジェクトのコードを読み解けない」「Next.jsやStorybookなど実務ツールを触ったことがない」── React/DB/Gitの基礎力と実務の間にある"最後のギャップ"が埋まっていない状態。
- **To be（あるべき姿）**：コース修了後、**Next.jsとReactを理解し、自分でAPIを作り、DBと連携してCRUD処理を実装し、Gitでプッシュできる**状態。これまで個別に学んだReact・PostgreSQL・Gitの知識を統合し、API連携 → DB操作 → Git運用 → Next.js＋Storybookでの開発を一気通貫で実践できる。
- **Specific（具体的）**：Next.js＋Storybookを用いたCRUD機能付きユーザー管理Webアプリを、開発の段取り → 実装 → 動作確認 → 整理のサイクルを回しながら構築できるようになる。
- **Measurable（測定可能）**：各Chapterの実践課題完遂率100%、確認試験の1発合格率80%以上・2回目合格率100%、想定学習時間内にコースを終えた受講者の割合が80%以上
- **Achievable（達成可能）**：「HTML・CSS初歩コース」「Reactではじめるプログラミング基礎コース」「データベースコース」「Git入門コース」修了者を対象に、これらの知識を統合しながら実務レベルの開発力を積み上げるカリキュラム
- **Relevant（関連性がある）**：プロジェクト参画時にCRUD処理やAPI連携のタスクを見て面食らわずに着手できる状態を作る。実務で初めてCRUDを見て手が止まるという現実の課題を事前に解消し、即戦力人材の育成に直結
- **Time-bound（期限がある）**：想定学習期間 約48〜64時間（1日2時間ペースで約1〜1.5ヶ月、集中取り組みなら6〜8日で完了可能）

# WHY - なぜ必要か

- **React・DB・Gitを"個別に学んだ" ≠ "仕事ができる"。** 各基礎コースでWebアプリ作成・DB操作・バージョン管理を個別に習得している。しかし現場では、これらを組み合わせて一つのプロダクトを作る技術力が求められる。**個々のスキルを"技術として統合する"最後のピースを埋めるコース**が必要。
- **実務で初めてCRUD処理やAPI連携を見て手が止まるという現実の課題がある。** プロジェクトに参画した際、CRUD処理を知らないとタスクに着手できない。コースの中で実際にCRUDを体験しておくことで、現場で初めて見たときに面食らわないようにする。
- **既存のオフライン研修（2週間）の技術的内容**を自己学習で再現可能な形に体系化し、いつでも・自分のペースで学べる環境を実現する。

# WHAT - 何をやるのか / HOW - 企画概要

- **4つの基礎コースの知識を"技術として統合する"ステップアップコース。** HTML/CSS・React・PostgreSQL・Gitを個別に学んだ受講者が、これらを組み合わせて実務レベルのWebアプリを開発するまでを一気通貫で体験する。
- **技術の統合と開発プロセスの同時習得設計。** 各Chapterで新しい技術を学ぶたびに、「開発の段取り」「実装」「動作確認」「実装結果の整理」を繰り返し練習するサイクルを組み込む。
- **Chapter1でコース概要と開発の段取りの考え方をインストール。** Chrome DevTools等のツールは独立した「使い方講座」としてではなく、Ch3以降の実装・デバッグの流れの中で自然に導入する。
- **差分確認はVSCode標準のGit機能（ソース管理タブ）を活用。** Git入門コースでCLI操作は習得済みのため、本コースではVSCode標準のソース管理タブを使い、コード差分の確認や不要コードの検知を習慣づける。自己学習ではテキストに沿って進めるためGit操作でのエラーは起きにくく、GUIツールの導入よりも「変更内容を目視確認する習慣」の定着を重視する。リポジトリは受講者自身のGitHubアカウント内で完結させる。
- **プロジェクトの配布はZIPファイル形式。** Git経由の配布で起こりうるアクシデント（masterブランチの誤更新等）を防ぐため、受講者はZIPをダウンロード → 展開 → 自分のGitHubリポジトリとして初期化する流れで進める。
- **DB操作はローカルPostgreSQLを使用。** データベースコースで習得したPostgreSQL/pgAdminの知識をそのまま活かし、ローカル環境でCRUD処理を実践する。クラウドサービスへの依存をなくし、環境構築のハードルを下げる。
- **最終課題「ユーザー管理システム」は実務シミュレーション。** ZIPから展開したプロジェクトを自分のGitHubリポジトリに登録 → コードリーディング → 開発の段取り → 実装 → VSCodeのソース管理タブで差分確認・動作確認 → 実装結果の整理のフルサイクルを繰り返し体験する。

# 想定質問と回答

- 前提コースを受講していないが参加できるか？
  - 本コースは以下の4コース修了が前提です：①「HTML・CSS初歩コース」（HTML/CSSの基礎）、②「Reactではじめるプログラミング基礎コース」（TypeScript/JSX/コンポーネント/state管理）、③「データベースコース」（PostgreSQL/pgAdmin操作）、④「Git入門コース」（CLIでの一連のGit操作）。未修了の場合は先にこれらのコースを受講してください。

- オフライン研修と内容は同じか？
  - 技術的な学習内容は研修と同等のものを提供していますが、大きく2点変更しています。①Supabase → ローカルPostgreSQLに変更（データベースコースの知識を直接活用）、②差分確認にVSCode標準のソース管理タブを活用し不要コード検知の習慣化を追加。基礎コースで習得済みのHTML/CSS・React基礎・DB基礎・Git CLI基礎はスキップしています。

- なぜGitではなくZIPでプロジェクトを配布するのか？
  - Gitでの配布（clone/fork）はmasterブランチの誤更新やリモートリポジトリへの意図しない操作といったリスクがあります。不特定多数が利用できる環境において、メンテナンス等はコストになるためZIPファイルで配布し、受講者自身のGitHubアカウント内で新規リポジトリとして管理してもらうことで、安全かつ実務に近い環境構築を体験できます。

- 講師への質問ができない環境で詰まった場合はどうするか？
  - SLスタジオ内のFAQにて非同期にはなりますが、質問対応予定です。
  - 研修として実施を行う場合は、セミナー等で「エラー対応5ステップ」と「10分ルール（10分調べてわからなければ質問）」を教えます。
  - 自己解決支援として、AIに対して質問する際のテンプレートの用意や「よくあるエラーと対処法」を用意します。

- このコースを終えると何ができるようになるのか？
  - Next.jsとReactを理解し、自分でAPIを作り、DBと連携してCRUD処理を実装し、Gitでプッシュできる状態になります。プロジェクト参画時にCRUD処理やAPI連携のタスクを見て面食らわずに着手できることを目指します。なお、報連相やビジネスマナーといったヒューマンスキルはコンセプトに記載の通りスコープ外です。

- 学習時間はどれくらい必要か？
  - 全体で約48〜64時間を想定しています。オフライン研修（2週間・約80時間）の技術的内容を自己学習用に再構成し、非技術領域をスコープ外とした上で、講師の説明パートを自分で読み進める形式のため、このボリュームになります。1日2時間ペースなら約1〜1.5ヶ月、集中取り組みなら6〜8日で完了可能です。

- 開発の段取りの演習は誰がフィードバックするのか？
  - SLスタジオ上では模範例との自己比較（セルフチェック）を基本とします。コース内に模範解答例を充実させ、受講者自身が客観的に評価できるようにします。


# カリキュラム構成案および各Chapterのゴール

- Chapter1：コースガイダンス ～ 開発の段取りを学ぶ
  - **As is（現状）**：タスクを受け取っても何から始めればいいかわからない。
  - **To be（あるべき姿）**：コース全体の学習サイクルを理解し、開発の段取り（設計書読解→実装順序の整理）の考え方を理解している。
  - **学べる技術セット**：
    - 開発の段取り（設計書読解・実装順序の整理）

- Chapter2：Webの仕組みとWeb APIを理解しよう
  - **As is（現状）**：Reactで画面は作れるが、データがどこからどう来るのか、リクエスト・レスポンスの仕組みが曖昧。APIドキュメントを見ても読み方がわからない。
  - **To be（あるべき姿）**：HTTPメソッド（GET/POST/PUT/DELETE）の役割を理解し、APIドキュメントを読んで必要な情報（URL・メソッド・パラメータ・レスポンス形式）を読み取れる。Thunder Clientを使ってAPIの疎通確認が自力でできる。
  - **学べる技術セット**：
    - HTTPメソッド（GET/POST/PUT/DELETE）
    - ステータスコード
    - JSON形式
    - APIドキュメントの読み方
    - Thunder Client

- Chapter3：ReactでWeb APIを叩いてみよう
  - **As is（現状）**：APIの仕組みは理解したが、Reactから実際にAPIを叩いてデータを取得・表示した経験がない。
  - **To be（あるべき姿）**：ReactのfetchでAPIを呼び出し、取得したJSONデータをstateで管理してUIに表示できる。DevToolsのConsole・Networkタブを使ってデバッグができる。API実装タスクの段取りを整理し、想定時間内に終わらせる段取りを組める。
  - **学べる技術セット**：
    - API
    - useState / useEffect
    - APIレスポンスのハンドリング
    - Chrome DevTools（Console・Networkタブ）
    - 開発の段取りの実践

- Chapter4：ReactとPostgreSQLを繋げよう（CRUD実践）
  - **As is（現状）**：pgAdminでSQLは書けるが、ReactからDBに接続してCRUD操作を行った経験がない。フロントエンドとデータベースが頭の中で繋がっていない。
  - **To be（あるべき姿）**：ReactからAPIを呼び出してCRUD操作を実装できる。データの登録・取得・更新・削除を一通り実装できる。フロントエンド → API → DB の全体アーキテクチャを理解している。
  - **学べる技術セット**：
    - CRUD操作（Create / Read / Update / Delete）
    - PostgreSQL連携
    - フロントエンド → API → DB のアーキテクチャ理解

- Chapter5：ダッシュボード開発で開発プロセスを実践しよう（総合演習）
  - **As is（現状）**：既存プロジェクトのコードを渡されても、どこから読めばいいかわからない。React/API/DBの知識はあるが、それらを組み合わせたプロジェクトをゼロから自力で進めた経験がない。
  - **To be（あるべき姿）**：VSCodeのソース管理タブでコード差分を確認しながら開発できる。ZIPから展開したプロジェクトをコードリーディングで把握し、開発の段取り→実装→動作確認→実装結果の整理のサイクルでダッシュボード画面を実装できる。
  - **学べる技術セット**：
    - コードリーディング
    - VSCodeソース管理タブでの差分確認
    - Gitブランチ運用
    - 開発プロセス（段取り → 実装 → 確認 → 整理）の実践

- Chapter6：Next.js＋Storybookに入門しよう
  - **As is（現状）**：Reactは書けるが、Next.jsのApp Routerやファイルベースルーティングに触れたことがない。Storybookの存在を知らない。
  - **To be（あるべき姿）**：Next.jsとReactの違い、App Routerによるルーティングの仕組みを理解している。Storybookの役割と活用場面を理解している。実際のプロジェクトでの活用はChapter7で実践する。
  - **学べる技術セット**：
    - Next.js（App Router）

  - **As is（現状）**：Next.js＋Storybookの基礎は理解したが、実務に近いプロジェクトでタスクベースの開発を一連の流れで進めた経験がない。
  - **To be（あるべき姿）**：既存プロジェクトをコードリーディングで把握し、新規登録・編集機能を開発の段取り→実装→動作確認→実装結果の整理のサイクルで自力で遂行できる。
  - **学べる技術セット**：

  - **As is（現状）**：機能は一通り動くが、再利用可能なコンポーネント設計やUI/UX改善の経験がない。
