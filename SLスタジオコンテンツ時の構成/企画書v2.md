# 企画タイトル

- 【コース企画】即戦力フロントエンド開発コース（React＋Next.js＋PostgreSQL）

# ゴール(SMART)

- **As is（現状）**：「HTML・CSS初歩コース」でWebの基礎を学び、「Reactではじめるプログラミング基礎コース」でWebアプリが作れるようになり、「データベースコース」でPostgreSQLの操作を学び、「Git入門コース」でバージョン管理のCLI操作を習得した。しかしこれらは個別の点でしかなく、実務で一人称として動くには"点と点を繋ぐ力"が足りていない。「ReactからAPIを叩いたことがない」「DBとフロントエンドを繋いだ経験がない」「既存プロジェクトのコードを読み解けない」「タスクの分解ができない」「質問や報告の仕方がわからない」「Next.jsやStorybookなど実務ツールを触ったことがない」── React/DB/Gitの基礎力と実務の間にある"最後のギャップ"が埋まっていない状態。
- **To be（あるべき姿）**：コース修了後、フロントエンドエンジニアとして**一人称**で仕事を進められる状態。すなわち、独力で全てを完結させるのではなく、**タスクを受け取ったら自分で言語化・分解し、提示された想定時間内に完了させるための段取りを組み、不明点は適切なフォーマットで質問・相談しながら、自走して成果物を完成させられる**人材になること。これまで個別に学んだReact・PostgreSQL・Gitの知識を統合し、API連携 → DB操作 → Git運用 → Next.js＋Storybookでの開発を一気通貫で実践。"学んだ"から"仕事として使える"へ進化する。
- **Specific（具体的）**：Next.js＋Storybookを用いたCRUD機能付きユーザー管理Webアプリを、タスク分解 → 想定時間内に収める段取り → 実装 → 報告のサイクルを回しながら構築できるようになる。不明点は適切に言語化し質問・相談ができる。
- **Measurable（測定可能）**：各Chapter確認試験・卒業試験の1発合格率80%以上、2回目合格率100%
- **Achievable（達成可能）**：「HTML・CSS初歩コース」「Reactではじめるプログラミング基礎コース」「データベースコース」「Git入門コース」修了者を対象に、これらの知識を統合しながら仕事力を積み上げるカリキュラム
- **Relevant（関連性がある）**：プロジェクト参画初日からタスクを受け取り一人称で動ける即戦力人材の育成に直結
- **Time-bound（期限がある）**：想定学習期間 約50〜60時間（1日2時間ペースで約1〜1.5ヶ月、集中取り組みなら3週間前後で完了可能）

# WHY - なぜ必要か

- **React・DB・Gitを"個別に学んだ" ≠ "仕事ができる"。** 「HTML・CSS初歩コース」「Reactではじめるプログラミング基礎コース」でWebアプリが作れるようになり、「データベースコース」でPostgreSQLを操作でき、「Git入門コース」でバージョン管理もできる。しかし現場では、これらを組み合わせて一つのプロダクトを作り、チームの一員としてタスクを自走する力が求められる。**個々のスキルを"仕事として統合する"最後のピースを埋めるコース**が必要。
- **一人称で動ける人材の育成が急務。** プロジェクト参画後に「何をすればいいですか？」と常に指示待ちの状態では、チームの生産性を損なう。タスクを自分で咀嚼し、わからないことは自ら調べ、それでも解決できなければ適切な形で質問・相談する ── この"一人称の仕事の進め方"を技術の統合学習と一体で身につける場が必要。
- **既存のオフライン研修（2週間）で講師がリアルタイム指導していた"仕事力"**（言語化フォーマット・報連相の型・タスク分解と段取り・エラー対応プロセス）を、自己学習で再現可能な形に体系化し、いつでも・自分のペースで学べる環境を実現する。

# WHO - ターゲットは誰か

- SLスタジオの以下4コースを修了済みの社員
  - 「HTML・CSS初歩コース」（HTML/CSSの基礎を習得済み）
  - 「Reactではじめるプログラミング基礎コース」（TypeScript/JSX/コンポーネント/state管理を習得済み）
  - 「データベースコース」（PostgreSQL/pgAdmin操作を習得済み）
  - 「Git入門コース」（CLI操作による一連のGitワークフローを習得済み）
- 個別の技術は学んだが、それらを統合して実務で使った経験がない方
- 「仕事の進め方」（タスク分解・時間管理・報連相・コードリーディング）に不安がある方
- フロントエンドエンジニアとしてプロジェクト参画を控えている社員

# WHAT - 何をやるのか / HOW - 企画概要

- **3つの基礎コースの知識を"仕事として統合する"ステップアップコース。** React・PostgreSQL・Gitを個別に学んだ受講者が、これらを組み合わせて実務レベルのWebアプリを開発するまでを一気通貫で体験する。
- **技術と仕事力の"二軸同時学習"設計。** 各Chapterで新しい技術を学ぶたびに、「タスクの言語化」「想定時間に対するタスク分解」「質問・相談のフォーマット」「完了報告」を繰り返し練習するサイクルを組み込む。
- **Chapter1で"仕事の型"を最初にインストール。** 研修で講師が口頭で教えていた以下の要素をLesson化し、コースの冒頭で仕事の進め方のフレームワークを身につける：
  - **タスクの言語化**：指示を受けたら概要・期限・作業時間・詳細・確認事項に分解するフォーマット
  - **質問・報連相のフォーマット**：概要・現在の状況・調査内容を構造化して伝える技術
  - **タスク分解と時間管理**：提示された想定時間内にタスクを完了させるために、作業を適切な粒度に分解し段取りを組む力
  - **エラー対応プロセス**：エラーを読む → 検索する → 該当箇所を探す → 対応策を調査 → 試す → 10分で解決できなければ質問する
  - **Chrome DevTools**：デバッグとUI確認に不可欠な開発者ツールの活用方法
- **差分確認はGitLens（VSCode拡張）を活用。** Git入門コースでCLI操作は習得済みのため、本コースではGitLens等のVSCode拡張を導入し、コード差分の確認や不要コードの検知を習慣づける。自己学習ではテキストに沿って進めるためGit操作でのエラーは起きにくく、GUIツールの導入よりも「変更内容を目視確認する習慣」の定着を重視する。リポジトリは受講者自身のGitHubアカウント内で完結させる。
- **プロジェクトの配布はZIPファイル形式。** Git経由の配布で起こりうるアクシデント（masterブランチの誤更新等）を防ぐため、受講者はZIPをダウンロード → 展開 → 自分のGitHubリポジトリとして初期化する流れで進める。
- **DB操作はローカルPostgreSQLを使用。** データベースコースで習得したPostgreSQL/pgAdminの知識をそのまま活かし、ローカル環境でCRUD処理を実践する。クラウドサービスへの依存をなくし、環境構築のハードルを下げる。
- **最終課題「ユーザー管理システム」は実務シミュレーション。** ZIPから展開したプロジェクトを自分のGitHubリポジトリに登録 → コードリーディング → タスク分解（言語化）→ 実装 → GitLensで差分確認・動作確認 → 振り返りのフルサイクルを繰り返し体験する。

# 想定質問と回答

- 前提コースを受講していないが参加できるか？
  - 本コースは以下の4コース修了が前提です：①「HTML・CSS初歩コース」（HTML/CSSの基礎）、②「Reactではじめるプログラミング基礎コース」（TypeScript/JSX/コンポーネント/state管理）、③「データベースコース」（PostgreSQL/pgAdmin操作）、④「Git入門コース」（CLIでの一連のGit操作）。未修了の場合は先にこれらのコースを受講してください。

- オフライン研修と内容は同じか？
  - 技術的な学習内容は研修と同等のものを提供していますが、大きく3点変更しています。①講師が口頭で指導していた「仕事の進め方」を明示的にLesson化、②Supabase → ローカルPostgreSQLに変更（データベースコースの知識を直接活用）、③差分確認にGitLens（VSCode拡張）を導入し不要コード検知の習慣化を追加。基礎コースで習得済みのHTML/CSS・React基礎・DB基礎・Git CLI基礎はスキップしています。

- なぜGitではなくZIPでプロジェクトを配布するのか？
  - Gitでの配布（clone/fork）はmasterブランチの誤更新やリモートリポジトリへの意図しない操作といったリスクがあります。不特定多数が利用できる環境において、メンテナンス等はコストになるためZIPファイルで配布し、受講者自身のGitHubアカウント内で新規リポジトリとして管理してもらうことで、安全かつ実務に近い環境構築を体験できます。

- なぜGitLensを使うのか？
  - Git入門コースでCLI操作は習得済みですが、実務では「変更した箇所に不要なコードが含まれていないか」を目視確認する習慣が重要です。GitLens（VSCode拡張）を導入し、エディタ上で差分を確認しながら開発する習慣を身につけます。自己学習ではテキストに沿って進めるためGit操作でのエラーは起きにくく、専用GUIツールよりもエディタ内で完結する差分確認の方が実用的です。

- 講師への質問ができない環境で詰まった場合はどうするか？
  - コース内で「エラー対応5ステップ」と「10分ルール（10分調べてわからなければ質問）」を教えます。各Lessonに「よくあるエラーと対処法」「つまずきポイントFAQ」やAIに対して質問をする際のテンプレートを用意し自己解決を支援します。なお解決困難な場合はSLスタジオのFAQにて質問をしてもらう想定です。

- 「一人称で仕事ができる」とはどういう意味か？
  - 独力で全てを完結させることではありません。タスクを受け取ったら①自分で内容を分解・言語化し、②提示された想定時間に対して段取りを組み、③不明点は適切なフォーマットで質問・相談し、④実装を進め、⑤完了したら報告する ── このサイクルを自ら回せる状態を指します。

- 学習時間はどれくらい必要か？
  - 全体で約50〜60時間を想定しています。オフライン研修（2週間・約80時間）の内容を自己学習用に再構成しており、講師の説明パートを自分で読み進める形式のため、このボリュームになります。1日2時間ペースなら約1〜1.5ヶ月、集中取り組みなら3週間前後で完了可能です。

- 言語化演習は誰がフィードバックするのか？
  - SLスタジオ上では模範例との自己比較（セルフチェック）を基本とします。コース内に評価ルーブリックと模範解答例を充実させ、受講者自身が言語化レベルを客観的に評価できるようにします。

- AI活用に関するLessonは必須か？
  - AI活用（GitHub Copilot・ChatGPT・Claude等）のLessonは**任意（オプション）**です。利用者や所属企業のAI利用方針が異なるため、コースの修了要件には含めていません。AI活用が許可されている環境の方は、開発効率向上の参考としてご活用ください。

# 各Chapter終了時点の達成度

## Chapter1：一人称エンジニアの"仕事の型"

- **As is（現状）**：タスクを受け取っても何から始めればいいかわからない。質問したいが何がわからないかを整理できない。エラーが出たら固まってしまう。
- **To be（あるべき姿）**：タスクを受け取ったら言語化フォーマットで分解し、提示された想定時間に対して段取りを組める。質問・報連相のフォーマットを使って不明点を構造的に伝えられる。エラーが出ても5ステップで自己対応を試み、10分で判断して質問に切り替えられる。DevToolsでUIとネットワークの確認ができる。

## Chapter2：Webの仕組みとWeb API

- **As is（現状）**：Reactで画面は作れるが、データがどこからどう来るのか、リクエスト・レスポンスの仕組みが曖昧。APIドキュメントを見ても読み方がわからない。
- **To be（あるべき姿）**：HTTPメソッド（GET/POST/PUT/DELETE）の役割を理解し、APIドキュメントを読んで必要な情報（URL・メソッド・パラメータ・レスポンス形式）を読み取れる。Thunder Clientを使ってAPIの疎通確認が自力でできる。

## Chapter3：ReactでWeb APIを叩いてみよう

- **As is（現状）**：APIの仕組みは理解したが、Reactから実際にAPIを叩いてデータを取得・表示した経験がない。
- **To be（あるべき姿）**：ReactのfetchでAPIを呼び出し、取得したJSONデータをstateで管理してUIに表示できる。API実装タスクを言語化フォーマットで分解し、想定時間内に終わらせる段取りを組める。

## Chapter4：ReactとPostgreSQLを繋げよう（CRUD実践）

- **As is（現状）**：pgAdminでSQLは書けるが、ReactからDBに接続してCRUD操作を行った経験がない。フロントエンドとデータベースが頭の中で繋がっていない。
- **To be（あるべき姿）**：ReactからAPIを呼び出してCRUD操作を実装できる。データの登録・取得・更新・削除を一通り実装できる。フロントエンド → API → DB の全体アーキテクチャを理解している。

## Chapter5：ダッシュボード開発（総合演習）

- **As is（現状）**：既存プロジェクトのコードを渡されても、どこから読めばいいかわからない。React/API/DBの知識はあるが、それらを組み合わせたプロジェクトを一人で進めた経験がない。
- **To be（あるべき姿）**：GitLensでコード差分を確認しながら開発できる。ZIPから展開したプロジェクトをコードリーディングで把握し、タスク分解（言語化）→実装→動作確認→振り返りのサイクルでダッシュボード画面を実装できる。

## Chapter6：Next.js＋Storybook入門

- **As is（現状）**：Reactは書けるが、Next.jsのApp Routerやファイルベースルーティングに触れたことがない。Storybookの存在を知らない。
- **To be（あるべき姿）**：Next.jsプロジェクトの構造を理解し、App Routerによるルーティングの仕組みがわかる。Storybookでコンポーネント単体の表示確認・ドキュメント化ができる。既存のNext.jsプロジェクトのコードを読み解ける。

## Chapter7：ユーザー管理システム①（新規登録・編集機能）

- **As is（現状）**：Next.js＋Storybookの基礎は理解したが、実務に近いプロジェクトでタスクベースの開発を一人称で進めた経験がない。
- **To be（あるべき姿）**：既存プロジェクトをコードリーディングで把握し、新規登録・編集機能をタスク分解（言語化）→ 実装 → 動作確認 → 振り返りのサイクルで一人称で遂行できる。

## Chapter8：ユーザー管理システム②（一覧・削除・詳細機能）

- **As is（現状）**：新規登録・編集は実装できたが、一覧表示・削除・詳細表示といった異なるパターンのCRUD実装経験が不足している。仕様が曖昧なタスクへの対応力がない。
- **To be（あるべき姿）**：一覧・削除・詳細の各機能を一人称で実装できる。仕様が曖昧なタスクに対して、自ら確認事項を洗い出し質問フォーマットで相談できる。動的ルーティングを活用した画面遷移を実装できる。

## Chapter9：ユーザー管理システム③（UI改善とコンポーネント拡充）

- **As is（現状）**：機能は一通り動くが、再利用可能なコンポーネント設計やUI/UX改善の経験がない。Storybookの活用が基本レベルにとどまっている。
- **To be（あるべき姿）**：共通ボタン・カード・モーダルなど再利用可能なコンポーネントを設計・実装し、Storybookで複数バリエーションをドキュメント化できる。アプリケーション全体のUIを統合・改善し、完成した成果物を言語化レポートとしてまとめられる。

## Chapter10：一人称エンジニアとして現場に出よう（総括）

- **As is（現状）**：コースを通じて技術と仕事力を身につけたが、実務の現場でどう動けばいいか具体的なイメージがまだ曖昧。
- **To be（あるべき姿）**：実装プロセスのベストプラクティスを理解し、プロジェクト参画時の動き方・心構えを把握している。セルフチェックリストで自身の一人称エンジニアとしての到達度を客観的に評価でき、現場に出る準備が整っている。

## Chapter11：卒業試験

- **As is（現状）**：各Chapterの確認試験は通過しているが、コース全体を通した技術・仕事力の総合的な理解度が未検証。
- **To be（あるべき姿）**：Web API・CRUD・Git運用・Next.js＋Storybook・タスク言語化・報連相・タスク分解と段取りなど、コースで学んだ全領域の知識を横断的に問われても回答でき、一人称エンジニアとしての総合力が証明された状態。
