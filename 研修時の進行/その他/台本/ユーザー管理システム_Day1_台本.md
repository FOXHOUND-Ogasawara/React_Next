# ユーザー管理システム_Day1_台本

皆さん、こんにちは！前回のダッシュボード画面の開発に関する資料に続き、今回は「ユーザー管理システム_day1.pdf」**という資料を見ていきましょう。**

**この資料は、これから皆さんが取り組む**ユーザー管理システム開発の初日**に行うこと、特に**新規登録機能**と**ユーザー編集機能**の実装に焦点を当てています。

まず、開発を始めるための**プロジェクトの準備**から説明されています。

ステップとしては、まず展開されたプロジェクトから「フォーク」を行い、

自分用のプロジェクトのコピーを作成します。

次に、作成したプロジェクトをローカル環境に**クローン**します。具体的には `$ git clone https://~` というコマンドを実行し、その後 `$ cd user-management-system` でプロジェクトディレクトリに移動します。

プロジェクトの準備ができたら、必要な**依存関係をインストール**します。これは `$ npm install` というコマンドを実行することで行います。

さらに、**環境用のファイルを作成**する必要があります。.env.localというファイルをプロジェクトの直下に作成し、以下の内容を追記します。
`NEXT_PUBLIC_SUPABASE_URL=https://your-supabase-url.supabase.coNEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-keyURL`と`ANON_KEY`については、別途Slackで共有されたものを利用するよう指示されています。

これらの手順が完了したら、プログラムが実行できるか**動作確認**をします。`$ npm run dev` コマンドを実行して確認できるとのことです。また、Storybookを確認するには `$ npm run storybook` コマンドを使用します。

さて、本プロジェクトについてですが、お題は**ユーザー管理システムを模したアプリケーション**となります。これまでの研修で学んだ**API通信やCRUD処理**を活かして、コンポーネントを実装していくことになります。

このプロジェクトを進める上で、以下の点が意識すべきポイントとして挙げられています:

- あくまでも**Reactを用いたプロジェクト**であること。文法やルールが一部異なりますが、Reactの知識が基盤となります。
- **これまでとは異なる実装方法**が含まれていること。**Next.js**や**Storybook**という新たな技術を利用します。確認や質問を怠らないことが重要です。
- 各タスクを完了したら、**リモートリポジトリにプッシュ**すること。

特に大きく異なるポイントとして、**ルーティング方法**が挙げられています。

Next.jsの**App Router**という技術を利用しており、`app` 配下に `page.tsx` を作成することで**自動的にルーティング**される仕組みです。

また、**APIを利用した通信処理**では、`fetch` を使わずに簡潔な記述が可能で、ファイル内のどこかに参考になる記述があるとのことです。

ここで新たな技術である**Storybook**について説明されています。

`components` 配下に `コンポーネント名.stories.tsx` というファイルが存在しており、これを利用します。

これまでコンポーネント単体での表示確認は難しかったですが、Storybookを使うことで

**コンポーネント単体での確認やテストが可能**となり、さらには**使い方のドキュメント化**もできる

便利な仕組みです。

最初は難しく感じるかもしれませんが、基本的にサンプルを真似て作ることで実装できるため、

まずは「コンポーネントを単体で確認できる機能」として捉え、作り方やルールを覚えるようにしましょう。

タスクに着手する前に、**コードリーディング**を行うことの重要性も説かれています。

実務では既存プロジェクトに参画することが多いため、ソースコードを読み解く能力が求められるからです。

クローンしたプロジェクトを確認し、質問や確認事項を事前に洗い出すことで効率良く作業を進められます。

全てを把握する必要はなく、**トップページ、ユーザー一覧ページ、ユーザー編集ページ**の内容がどこに記載されているかを重点的に確認し、関連ファイルを探す流れで進めると良いでしょう。

本プロジェクトで取り組む**タスク**は、大きく分けて以下の2つです:

1. **新規登録機能の実装**:
    - コンポーネント作成
    - ストーリーブック作成
    - 画面作成
2. **ユーザー編集機能の実装**:
    - コンポーネント作成
    - ストーリーブック作成
    - 画面作成

これらのタスクについて、さらに詳細な説明があります。

まず、**新規登録機能関連のタスク**です。

**タスク1-1：新規登録画面コンポーネントの作成**
これは、ユーザーの新規登録用フォームコンポーネントを作成するタスクです。対象ファイルは `components/RegisterForm.tsx` で、そこに資料に示されたソースコードをコピーして

作業を行います。
仕様として、コンポーネント名が `RegisterForm` であること、`RegisterFormProps` で定義した内容を受け取れること、名前、メール、ロールの登録機能が実装されていること、

「登録」ボタンで登録が可能であること、そして**登録成功後にユーザー一覧画面に遷移**されること が挙げられています。

遷移する仕組みについてはヒントを参考にします。
資料のコード片からは、`React.FC`、フォーム管理のための `react-hook-form`、UIコンポーネントとして `@mui/material` が利用されていること、そして新規登録処理として `../utils/api` からインポートされた `createUser` 関数が使われることがわかります。
ヒントとしては、データ登録成功時のページ遷移を関数としてPropsから受け取る方法が示されており、`RegisterFormProps` インターフェースの例として `onSuccess?`、`onError?`、`disabled?` が提示されています。

**タスク1-2：RegisterFormのストーリー作成**
これは、作成した `RegisterForm` コンポーネントのStorybookストーリーを作成し、コンポーネントのドキュメント化を行うタスクです。対象ファイルは `components/RegisterForm.stories.tsx` で、こちらも資料のソースコードをコピーして作業します。
仕様として、`Meta` を使用してストーリーメタデータを定義していること、`StoryObj` を使用してストーリーを定義していること、デフォルトストーリーが設定されていること、そして実行確認ができること が挙げられています。
ヒントとして、デフォルトストーリーで渡すデータは `RegisterForm` が求めるPropsであり、例えば登録完了時に関数を受け取るPropsには `alert();` などを設定することで、画面遷移以外の挙動を確認できると説明されています。

**タスク1-3：新規登録ページの作成**
これは、作成した `RegisterForm` コンポーネントを使用して新規登録画面を作成するタスクです。対象ファイルは `app/register/page.tsx` と `components/Navbar.tsx` です。
仕様として、`RegisterForm` コンポーネントをレンダリングし、新規登録画面が表示できること、ページ全体のレイアウトが崩れていないこと、**新規登録ページへのリンクが上部メニューに追加されている**こと、そしてタスク1-1で実装したユーザー一覧画面への遷移が実装されていること が挙げられています。また、正常に動作していることを確認します。
資料のコード片からは、Next.jsのApp Routerの仕組みでページを定義する `app/register/page.tsx` であること、クライアントコンポーネントとしてマークされていること (`'use client'`)、そして `RegisterForm` をインポートして使用することがわかります。

次に、**ユーザー編集画面関連のタスク**です。

**タスク2-1：ユーザー編集画面コンポーネントの作成**
これは、ユーザー情報を編集するフォームコンポーネントを作成するタスクです。対象ファイルは `components/EditUserForm.tsx` で、こちらも資料のソースコードをコピーして作業します。
仕様として、**`EditUserFormProps` インターフェースを定義**していること（必要なプロパティはソースコードで確認）、コンポーネント名が `EditUserForm` であること、`EditUserFormProps` で定義した内容を受け取れること、ユーザーの編集機能（名前、メール、ロールの編集）が実装されていること、「更新」ボタンでデータ更新が可能であること、そして**更新成功後にユーザー一覧画面に遷移**されること（遷移の仕組みはタスク1-1と同様） が挙げられています。
資料のコード片からは、クライアントコンポーネントとしてマークされていること (`'use client'`)、`useEffect`、`useForm`、`@mui/material`、ユーザー情報の取得・更新処理として `../utils/api` からインポートされた `fetchUserById` および `updateUser` 関数、そしてルーティングのために `next/navigation` からインポートされた `useRouter` が利用されることがわかります。また、コンポーネントはPropsとして `userId` を受け取ることが示されています。

**タスク2-2：EditUserFormのストーリー作成**
これは、作成した `EditUserForm` コンポーネントのStorybookストーリーを作成し、コンポーネントのドキュメント化を行うタスクです。対象ファイルは `components/EditUserForm.stories.tsx` です。
仕様として、`Meta` を使用していること、`StoryObj` を使用していること、そして**デフォルトストーリーに `userId` が「1」のユーザーが設定されている**こと が挙げられています。実行確認をすると、"invariant expected app router to be mounted" というエラーが発生することが示唆されています。
ヒントとしては、デフォルトのストーリーで渡すデータは `EditUserForm` が求めるPropsであり、更新完了時に関数を受け取るPropsには `alert();` などを設定することで、画面遷移以外の挙動を確認できると説明されています。

**タスク2-3：ユーザー編集ページの作成**
これは、作成した `EditUserForm` コンポーネントを使用してユーザー編集画面を作成するタスクです。対象ファイルは `app/users/[id]/edit/page.tsx` です。
仕様として、`EditUserForm` コンポーネントをレンダリングし、編集画面が表示されること、**対象となるユーザー情報がデフォルト値として設定されている**こと、ページ全体のレイアウトが崩れていないこと、そしてタスク2-1で実装したユーザー一覧画面への遷移が実装されていること が挙げられています。また、正常に動作していることを確認します。
資料のコード片からは、App RouterのDynamic Routesの仕組みでページを定義する `app/users/[id]/edit/page.tsx` であること、クライアントコンポーネントとしてマークされていること (`'use client'`)、`EditUserForm` をインポートすること、そして**URLパラメータからユーザーIDを取得するために `next/navigation` からインポートされた `useParams` を利用する**ことが示唆されています。

この資料は、プロジェクトの準備から、具体的な機能である新規登録とユーザー編集について、コンポーネント、Storybook、ページ作成という手順で詳細なタスク内容、対象ファイル、仕様、さらにはコード片やヒントまで具体的に示しています。この資料に沿って進めることで、効率的に開発を進めることができるでしょう。
