## GitHubアカウントの作成

下記のリンクにアクセスしましょう。

公式ページ：[https://github.com/](https://github.com/)

![image.png](image%2058.png)

トップページにある「Sing up」をクリックして、以下の必要な項目を入力してください。

- メールアドレス（FOX HOUNDドメインのもの）
- パスワード
- ユーザー名（本名をローマ字で）
- Githubの最新情報を受け取るか否か（不要ならチェックしない）
- アカウント保護（「検証する」をクリックして手順に従う）

すべての入力が完了したら「Create account」をクリックしてアカウントを作成しましょう。

登録したメールアドレスに認証用の数字（認証コード）が送られてくるので、指示に従って数字を入力しログインします。

Githubのダッシュボード画面が表示されたらアカウントの作成は完了です。

## GitBashの使い方

研修内ではVSCodeをメインで使うため、利用機会は少ないかもしれませんがGitBashの柄板についても簡単にまとめておきます。

### GitBashの構成要素

![image.png](image%2063.png)

①ユーザー名

自分のPCのユーザー名が表示されます。

②現在のディレクトリ階層

上部のヘッダー部分では、現在の階層がフルパスで表示されます。
下部の②のようにコマンド入力欄の末尾にも表示されます。

③コマンド

実際のコマンド部分です。
ここに様々なコマンドを入力して実行を行います。

④＄マーク

コマンド入力欄の先頭に表示される記号です。
ここにコマンドを入力してね、というイメージです。
入力するコマンドを表記する場面でも `$ cd xx` のように$マークを付ける習わしがあります。

### お作法

```bash
$ git コマンド名
```

Gitコマンドを入力する際は、初めに `git` と入力し、半角スペースを一つ空けつつコマンド名を入力することになります。

先頭の `$` はあくまでもコマンドであることを表すマークなので、自分では入力しません。
もう一度言います、 `$` は自分では入力しません。

### オプションと引数

```bash
$ git コマンド名 引数1 引数2 -オプション1 -オプション2
```

コマンド名の後ろに半角スペースを空けて、引数やオプションを付けることが可能です。

具体的な使い方は各コマンドで紹介するので、こういう書き方があるというのを覚えておいてください。

## Linuxコマンド

GitBashを使う上ためには、最低限のLinuxコマンドも覚えておく必要があります。
よく使うものを二つ記載しておくので、まずはこの二つを覚えておいてください。

### ディレクトリ階層の移動

```bash
$ cd 移動先のパス
```

Reactのプロジェクト作成時にも使っていたコマンドで、現在のディレクトリから指定したディレクトリに移動することが可能です。

```bash
$ cd 移動先のパス/移動先のパス

$ cd ../
```

cdコマンドで指定するパスは上記のように `/` を加えれば一気に移動することも可能です。
また、 `../` という記述は一つ上の階層に移動する際に利用します。

### ディレクトリ内の確認

```bash
$ ls
```

現在いる階層に保管されているファイルやディレクトリを一覧表示するコマンドです。
対象のファイルやディレクトリが存在するかを確認する場面で利用します。

```bash
$ ls パス

$ ls -a
```

lsコマンドは、パスを指定することで確認するディレクトリを変更したり、 `-a` オプションを付けることで隠しファイルなども含めて表示させることが可能です。

## Gitコマンドを使ってみよう

VSCodeを使って、Reactのプロジェクトを作ってみましょう。

ライブラリなどは何も入れず、シンプルな構成で問題ありません。

プロジェクト作成が完了したら、VSCodeのターミナルからGitコマンドが利用できるのかを確認します。

```bash
$ git --version
```

上手く動作しない人は、VSCodeのターミナル右側に表示されているターミナルが「powershell」になっていることを確認、再度コマンドを実行してみてください。

Gitが利用できることを確認したら以下の手順でGitによる管理を始めてみましょう。

### 初期設定

```bash
$ git init
```

上記コマンドを実行することで、 `Initialized empty Git repository in` というキーワードが表示され、Gitでの管理が開始されます。

Gitの管理を開始すると、 `.git` フォルダが追加されるのですが、設定によっては表示されませんので、以下のコマンドを実行して念のため確認してください。

```bash
$ ls -force
```

※実行しているpower shellの場合は `-force` オプションを付ける必要があります

![image.png](image%2064.png)

次に、ユーザー設定を行います。

```bash
$ git config --global user.email 自分のメールアドレス
$ git config --global user.name 自分の名前
```

自身のメールアドレスや名前を入力して登録しておきましょう。

名前は「名前-苗字」となるように登録し、メールアドレスは会社から割り振られたアドレスにしておきましょう。

登録された情報を確認する場合は、自分の○○部分を抜いたコマンドを実行することで確認することが可能です。

```bash
$ git config --global user.email

$ git config --global user.name
```

ユーザー情報の確認が取れたら、最後に以下のコマンドを実行しておいてください。

```bash
$ git config --global core.autoCRLF false
```

### ファイルの更新状況を管理する

まずは、以下のコマンドを実行してみましょう。

```bash
$ git add -A

$ git commit -m "first commit"
```

実行すると以下のような状態になるはずです。

![image.png](image%2065.png)

次に、以下のコマンドを実行してみましょう。

```bash
$ git branch
```

実行すると、 `master` というキーワードに色が付いた状態になるのが確認できます。

![image.png](image%2066.png)

今回入力した `branch` コマンドは、ブランチと読むのですが、作業する場所を一覧で表示したり、新しい作業場所を作成することができるコマンドです。

現時点では、 `master` と呼ばれる作業場所しかないということを表しているわけです。

では、試しに別の作業場所を作ってみましょう。

```bash
$ git branch sampleBranch
```

「sampleBranch」という名前で作業を追加することができたはずです。

確認してみましょう。

![image.png](image%2067.png)

次のコマンドは作業場所を変更するコマンドです。

```bash
$ git checkout sampleBranch
```

`checkout` の後ろにスペースを空けて、作業場所を指定します。

![image.png](image%2068.png)

先ほどまでとは異なり、「sampleBranch」に色とアスタリスクが付いているのが確認できます。

準備が完了したので、ソースコードの内容を修正しつつ、Gitコマンドを実行してみましょう。

＜src/App.tsx＞

```bash
function App() {
	return (
		<>
			<h1>sampleBranchのヘッダーです</h1>
		</>
		)
}
```

ファイルを修正すると、VSCode上部のファイル名が表示されている箇所に「M」と追記されている状態になるはずです。

この状態で、次のコマンドを実行してみましょう。

```bash
$ git status
```

ファイルなどに変更が加えられた状態で、 `status` コマンドを実行することで、差分が生じたファイル名などを確認することが可能です。

以下の画像では、「src/App.tsx」と記載されてるのがわかりますね。

![image.png](image%2069.png)

では、この変更した「状態」を履歴として保存しておきましょう。

実行するコマンドは先ほど一度打っているので難しくはないはずです。

```bash
$ git add -A

$ git commit -m "App.tsxを修正"
```

実行するとことで変更した状態（内容）が履歴として保存されました。

確認のために、 `log` コマンドを実行して履歴を表示してみましょう。

```bash
$ git log
```

![image.png](image%2070.png)

一番最初に保存した履歴の次に、App.tsxを修正したという履歴が残されていることが確認できれば正常にGitの管理が出来ている状態となります。

ここまでの流れを実際に体験しても「これがなんの役に立つのか？」という疑問が浮かぶのではないでしょうか？

では、VSCodeでApp.tsxを開いた状態で、もう一度 `master` に作業場所を切り替えてみましょう。

```bash
$ git checkout master
```

なにが起こったでしょうか？

App.tsxの内容が一瞬で修正する前に変化したのではないでしょうか？

これが、バージョン管理と呼ばれるシステムで、ファイルやフォルダなどを履歴ごとに保存して管理することで任意のタイミングで好きなように切り替えることができるのです。

<aside>
✅

【コマンド】

- `branch`
    - 作業場（ブランチ）の一覧や作成を行う
- `checkout`
    - 作業場（ブランチ）を切り替える
- `add -A`
    - 変更点を適応する
    `commit` とセットで使うことが多い
- `commit -m "コメント"`
    - 適応した変更点を履歴に保存する
    コミットするとも呼ぶ
- `status`
    - 変更したファイルなどを確認する
- `log`
    - 履歴を表示する

【キーワード】

- ブランチ
    - 作業場のことで、○○ブランチと呼ぶ
    タスクごとにブランチを作成して作業を進めることが多い
    ブランチを作成することを「ブランチを切る」と呼ぶ
- コミット
    - 変更内容を履歴として残すこと
    作業した内容をコミットするまでが一連の流れになっていく
- 履歴
    - コミット履歴のこと
</aside>

## GitHubを利用したリモートリポジトリ

### GitHub上にリポジトリを作成する

GitHubにログインし、インターネット上にリポジトリを作成しましょう。

この、インターネット上に存在するリポジトリのことを「リモートリポジトリ」と呼ぶので覚えておいてください。

画面上に表示される `Create repository` か `Start a project` のどちらかをクリックして進めていきましょう。

![image.png](image%2071.png)

リポジトリの作成画面まで進めると、リポジトリ名や公開設定を選択する項目が表示されます。

次の設定で作成してみましょう。

- リポジトリ名： `git-sample`
- 公開設定： `private`

他の設定は気にせず、下部の `Create repository` をクリックしましょう。

![image.png](image%2072.png)

作成すると、自動的にリポジトリのページに遷移します。

左上に `ユーザー名 / リポジトリ名` の形式で表示されていることを確認しておきましょう。

### リモートとローカルを紐づける

ひとつ前に作成した、Reactのプロジェクトと先ほど作成したリモートリポジトリを紐づけてみましょう。

VSCodeやGitBash上で、対象のプロジェクトを選択し、以下のコマンドを実行してみてください。

```bash
$ git remote add origin https://github.com/USER_NAME/PROJECT_NAME.git
```

`USER_NAME` と `PROJECT_NAME` の部分は、GitHubのリモートリポジトリ画面左上に表示されている名前を入力してください。

入力し、Enterを押すと、エラーが起こらなければなにも表示されません。

エラーがになっていない場合は、以下のコマンドを実行してください。

```bash
$ git push -u origin master
```

上記のコマンドを始めて実行するとGitHubとの認証を求める画面が表示されます。

`Sign in with your browser` をクリックして進めてください。

![image.png](image%2073.png)

ブラウザ上で上記のような画面が出るので、GitHubに登録したメールアドレスとパスワードを入力してサインインします。

すると、次のような画面が表示されるために `Authorize GitCredential Manager` をクリックします。

![image.png](image%2074.png)

紐づけが無事完了すれば、メッセージが表示されるので確認してみましょう。

![image.png](image%2075.png)

最後に、GitHub上に反映されているのかを確認するため、改めてGitHubにアクセスしてみてください。

リモートリポジトリを作成したタイミングでは、ファイルやデータはなにもない状態でしたが、自分で作成したReactのプロジェクトが反映されているはずです。

※反映されてない場合は、ブラウザを再読み込みしてみてください。

## Gitを使った作業を体験してみる

ローカル上のプロジェクト（リポジトリ）をリモートリポジトリと紐づけを行いましたが、この時点ではブランチは一つしか存在していません。

ローカル上で作業をした際に、もう一つのブランチを作成したはずですね。

GitHubの左上に表示されているプルダウンに `master` とありますが、クリックしても他のブランチは出てこないはずです。

### ローカル上のブランチを反映する

まず初めに、ローカル上に新しいブランチを切ってみましょう。

以前作成した `sampleBranch` に切り替えた後、 `task01` という名前でブランチを作成してください。

ブランチを作成後にチェックアウトで切り替え、以下のコマンドを実行します。

```bash
$ git push -u origin task01
```

すると、リモートリポジトリにも `task01` というブランチが追加されるはずです。

GitHub上で確認してみましょう。

![image.png](image%2077.png)

実務によってブランチ名の違いや、リモートリポジトリに紐づけるかどうかは変化しますが、基本的にはタスクを開始するタイミングで新しいブランチを作成して作業を行います。

では、 `task01` ブランチ上で作業を行い、それをリモートリポジトリにも反映してみましょう。

作業内容は以下の通り。

- 任意のユーザー情報を作成し、テーブル表示させる機能を追加する
- 作業が完了したタイミングでリモートリポジトリに反映する

リモートリポジトリに反映するためには、変更点をコミットする必要があります。

復習も兼ねて、作業内容をコミットしておきましょう。

コミットが完了したら、 `push` コマンドでリモートにプッシュを行います。

```bash
$ git push
```

プッシュすることで、リモートリポジトリに反映されるため、改めてGitHub上でも確認してみましょう。

### 作業内容を統合する

Gitを利用した作業は基本的に以下のような流れで行われます。

1. 元となるブランチから作業用のブランチを作成する
2. 作業用ブランチにてタスクに応じた作業を行う
3. 作業のチェックポイントの要領でコミットをしていく
4. すべての作業が終わったらリモートリポジトリにプッシュ
5. 他の作業者の内容を結合していく
6. リリース

Gitを使う利点として、リモート上で簡単に他の作業者のコミットを取り込むことが可能です。

複数人での開発を行う場合、作業箇所を分担し、タスクに切り分け、各々が作業を進めていきます。

そして、最後に各々の作業を一つに結合し、動作に問題がないかを確認し、最終的にはお客さんが見れる状態に反映されるのです。

では、今回は練習を兼ねて自身の作業内容のみを `master` ブランチに反映してみましょう。

GitHubの上部タブから `Pull requests` を選択し、 `New pull request` をクリックします。

`base: maset` ← `compare:xxx` となっている右側を `task01` に変更してみましょう。

![image.png](image%2078.png)

すると、 `master` に対して `task01` のコミット内容がどのように反映されるのかが表示されます。

この時、 `sampleBranch` を作成して作業した内容も含まれているので確認しておきましょう。

`Create pull request` をクリックすると、次のような画面に切り替わります。

この画面では、コメントを入力することができるため、任意のコメントを記載してみましょう。

![image.png](image%2079.png)

ここまでの作業が プルリクエスト です。

自分で行った作業内容を、他のレポジトリに対してマージしてくださいという依頼を出す作業です。

開発の現場では、複数人で作業を行うのですが、その中にはプロジェクトを束ねるリーダーが存在しています。

そのリーダーに対して、タスクの内容を反映してくださいと依頼し、確認後に取り込んでもらうことでそのタスクが完了となります。

ここからの作業は、リーダーが担うことが多いのですが、経験しておきましょう。

![image.png](image%2080.png)

プルリクエストを出すことで、作業内容やコミットメッセージ、プルリクエストのメッセージなどが表示されるようになります。

これらの情報を確認し、問題がなければ作業内容をマージしていくことになります。

`Merge pull request` を一度クリックすると、内容が変化し `Confirm merge` になります。

![image.png](image%2081.png)

さらに `Confirm merge` をクリックすることで、プルリクエストを承諾し、マージが行われます。

操作が完了すると、 `master` ブランチの内容が更新されます。

GitHub上から確認してみましょう。

### 差分を取り込む

先ほどのマージを行ったことで、ローカル上の `master` と リモート上の `master` に差が生じたことがわかります。

先ほどローカル→リモートの反映は行いましたね。次は逆に、リモート→ローカルの反映方法を体験してみましょう。

といっても簡単です。

```bash
$ git fetch

$ git merge
```

この二つのコマンドを順に実行するだけです。

`fetch` コマンドを実行すると、リモートとローカルの差分が取得でき、内容を確認することが可能です。

その後に、 `merge` コマンドを実行することで、ローカルに反映を実施します。

## チームでやってみる

誰か一人のGitHub上のプロジェクトを利用します。

代表の人は、GitHub上のプロジェクトにほかのメンバーを追加してあげましょう。

追加された人は、メールが届くので参加します。

参加したのち、GitHubにアクセスしてCloneするためにURLをコピーし、以下のコマンドを実行してください。

コマンドを実行したフォルダ内に直接プロジェクトが作成されます。
そのため、ドキュメントフォルダ内に `git_sample` という名前でフォルダを作成し、 `cd` コマンドで移動したうえで下記のコマンドを実行するようにしてください。
※わからずに実行すると後々面倒なことになります。
　不安な人は必ず講師に確認をおこなってください！

```bash
$ git clone URL
```

### 作業用のブランチを作ってみる

誰か一人が、新しいブランチを作成し、リモートにプッシュしてみましょう。

ほかの人は、ブランチをローカルに取り込みます。

- ブランチを新規作成する人は以下の手順を実行する

```bash
// developブランチを作成し選択する
$ git branch develop
$ git checkout develop

// 必要に応じてコミットする

// リモートへの初回プッシュ
$ git push -u origin develop

// 自分用の作業ブランチを作成
$ git branch develop_YOURNAME
$ git checkout develop_YOURNAME
```

- 他のメンバーは以下の手順を実行する

```bash
// ローカルへブランチを反映
$ git fetch
$ git checkout -b develop origin/develop

// 作業用のブランチを作成
$ git branch develop_YOURNAME
$ git checkout develop_YOURNAME
```

各自のローカル上で「App.tsx」を修正し、リモートに反映してください。
反映するためには、コミット⇒プッシュを行います。

反映後、 `develop` ブランチに対してプルリクエストを投げてみましょう。

今回はプロジェクトを共有した人がメインとなり、順番にプルリクエストを承諾しマージしてみてください。

二人目の人が投げたプルリクエストをマージしようとするとコンフリクトが発生し、以降のマージができなくなってしまうはずです。

### コンフリクトを修正する

同じファイル内で変更内容に差分が出ると **コンフリクト** が発生しソースコードの結合ができない状態になってしまいます。

コンフリクトが発生した場合は該当箇所を修正し、再度コミット⇒プッシュを行うまでプルリクエストが承認できないため、必ずこの作業を行う必要があります。

手順は以下の通りです。

```bash
// ベースのブランチを最新状態にする
$ git checkout develop
$ git fetch
$ git merge

// 作業ブランチに戻る
$ git checkout develop_YOURNAME

// 最新状態のベースのブランチを作業ブランチに取り込む
$ git merge develop
```

上記のコマンドを実行することでVSCODE上にコンフリクトした内容が表示されるようになります。

今回は「App.tsx」がコンフリクトしているため他メンバーと確認しながら修正しつつ、再度プッシュまでのコマンドを実行してください。

修正した内容をプッシュすることでGitHub上のコンフリクト表記が解消され、マージすることが可能となります。

### 変更のコミット、リモートリポジトリにプッシュするまで

続いて、ファイル内容を変更してコミットとリモートリポジトリにプッシュするまでの流れです。

VSCodeで`App.tsx`を以下のように書き換えてください。

全てコピペして問題ないです。

```tsx
import "./App.css";

function App() {
  return (
    <>
      <h1>mainのヘッダーです</h1>
    </>
  );
}

export default App;

```

コミット内容を**必ず**チェックしてから`Summary(required)`の箇所に「App.tsxの修正」とコメントを入力してコミットをしましょう。

ショートカットキーは、`Ctrl＋Enter` です。

![image.png](image%2097.png)

元の画面に戻ったら`Push origin`を押すことでリモートに今のコミット内容をプッシュすることが出来ます。

ショートカットキーは、`Ctrl＋P` です。

![image.png](image%2098.png)

<aside>
💡

コミット内容の確認を必ずしましょう。

コマンド操作の時と違い、特に操作をしなくても画面上で修正前と修正後を確認することが出来ます。

修正前：赤

修正後：緑

コミット前には必ず内容をチェックする癖を付けてください。

チェックするポイントは

- 不要なコメントが残っていないか
- 動作確認用のconsole.logが残っていないか
- 未使用の処理が記載されていないか
- 誤字脱字がないか（メソッド名や変数名など）
- 誤って保存した内容がないか

コミットした内容というのは最終的にPRを出して、上長がコードレビューをして問題なければマージをします。

コードレビュー時に不要なものが混ざっていると無駄な確認作業が必要になったり、一度破棄して再度PRを出す必要が出てしまいます。

必ずチェックをするようにしましょう。

</aside>

### プルリクエストを作成する

プルリクエストの作成はGithubからでも操作可能ですが、Github Desktopからプルリクエストの画面を立ち上げることができます。

画面上の「Create Pull Request」を押してください。

ショートカットキーは`Ctrl＋R`です。

![image.png](image%20102.png)

そうすると、Githubが立ち上がるのでそこからの手順は変わらないです。

マージ先とマージ元を確認してからプルリクエストを作成してください。

![image.png](image%20103.png)

この後、リモートの差分をローカルに取り込むのでプルリクエストの承認をしておいてください。

### 差分を取り込む

ブランチを`main`に切り替えます。

「Current Branch」を押して、`main`を選択します。

「Current Branch」を開くショートカットキーは`Ctrl+B`です。

![image.png](image%20104.png)

その後、「Fetch origin」を押します。

そうすると「Fetch origin」から「Pull origin」に変わるので押すことで`pull`が出来ます。

ショートカットキーは`Ctrl+Shift+T`でfetch、`Ctrl+Shift+P`でpullが出来ます。

pullはmergeと同義です。
